# Share-card feature ‚Äî 2026 verification brief (Expo SDK 57)

## 1. react-native-view-shot

**Version & compatibility ‚Äî VERIFIED**
- Current npm version: **5.1.1** (npm registry, `latest`); peer deps `react >= 18`, `react-native >= 0.76` ‚Äî satisfied by SDK 57 / RN 0.86. https://registry.npmjs.org/react-native-view-shot/latest
- **Expo SDK 57 pins 5.1.0** in `bundledNativeModules.json` (`"react-native-view-shot": "5.1.0"`) ‚Äî install with `npx expo install react-native-view-shot` and take the pinned version. https://github.com/expo/expo/blob/sdk-57/packages/expo/bundledNativeModules.json
- **No config plugin, no permissions.** It's a plain autolinked native module; the SDK 57 docs page lists it as included in Expo Go, and it works in dev builds. https://docs.expo.dev/versions/latest/sdk/captureRef/
- 5.x is a New-Architecture rewrite (Fabric + TurboModules, `codegenConfig` present); 4.0.3 was the prior line. Actively maintained through 2026. https://github.com/gre/react-native-view-shot/releases

**Capturing off-screen at exactly 1080x1920 ‚Äî VERIFIED WITH A CRITICAL NUANCE**
- `captureRef` options `width`/`height` **do resize the final image** ("the width and height of the final image (resized from the View bound)") ‚Äî the DESIGN.md claim is directionally correct. https://github.com/gre/react-native-view-shot
- **BUT the options are in logical points, not pixels.** Output pixel size = option value x device scale. The official Expo docs example computes `pixels = targetPixelCount / PixelRatio.get()` and passes that as `width`/`height`. So for a guaranteed 1080x1920 file on every device: `captureRef(ref, { width: 1080 / PixelRatio.get(), height: 1920 / PixelRatio.get(), format: 'png' /* or jpg */, quality: 1 })`. https://docs.expo.dev/versions/latest/sdk/captureRef/
- The resize is a plain scale, not aspect-fill ‚Äî the off-screen card view itself must be styled at a 9:16 aspect (e.g. 360x640 pt) or the output distorts.
- **Off-screen gotchas (README-documented):**
  - The view must be **mounted and laid out**: wait for at least one `onLayout` (or a timeout) before capturing; unmounted/`display:none` views cannot be captured (issues #242, #31). The accepted pattern is an absolutely-positioned view moved off-viewport or rendered behind the visible UI ‚Äî exactly what DESIGN.md intends.
  - **Android:** the target needs `collapsable={false}` or React may flatten it away and you get a blank image (using the library's `<ViewShot>` wrapper component handles this automatically).
  - Known blank-image causes on Android: capture before layout, very heavy child images, virtualized list children. https://github.com/gre/react-native-view-shot ; https://github.com/gre/react-native-view-shot/issues/198

**Better 2026 alternative? ‚Äî NO**
- There is no "expo-view-shot successor"; react-native-view-shot **is** the Expo-documented capture API in SDK 57.
- Skia (`@shopify/react-native-skia`) offers `makeImageFromView` / canvas `makeImageSnapshot`, but snapshot resolution is tied to screen pixel ratio (worse for a fixed 1080x1920 target), it has known Android issues capturing mixed RN/Skia content (Shopify/react-native-skia#1633), and true offscreen Skia drawing would require re-typesetting the card with Skia's text APIs. DESIGN.md's "Skia is unnecessary" call is confirmed. https://shopify.github.io/react-native-skia/docs/snapshotviews/ ; https://dev.to/dawidzawada/offscreen-drawing-with-react-native-skia-1m5e

## 2. expo-media-library + expo-sharing

**expo-media-library ‚Äî API CHANGED (correction needed)**
- SDK 57 bundles `expo-media-library ~57.0.1` with a **new class-based API, now stable**. `saveToLibraryAsync()` / `createAssetAsync()` are legacy ‚Äî replaced by **`Asset.create(localUri)`** (saves the file and returns an Asset; pass an album param to avoid a second user confirmation). The old functions remain importable from `expo-media-library/legacy` for now. https://docs.expo.dev/guides/sdk-libraries-migration/media-library/ ; https://docs.expo.dev/versions/latest/sdk/media-library/
- **iOS add-only: still real and correct.** Config plugin `savePhotosPermission` ‚Üí `NSPhotoLibraryAddUsageDescription`; `requestPermissionsAsync(writeOnly: true)` / `usePermissions({ writeOnly: true })` requests write-only access. Set `photosPermission: false` if you never read the library. (Plugin source: expo/expo `packages/expo-media-library/plugin/src/withMediaLibrary.ts`, sdk-57 branch.)
- **Android ‚Äî NEW RISK not in DESIGN.md:** the config plugin's `granularPermissions` option **defaults to `["photo","video","audio"]`**, which puts `READ_MEDIA_IMAGES`, `READ_MEDIA_VIDEO`, `READ_MEDIA_AUDIO` in the manifest. Google Play's **Photo and Video Permissions policy** (fully enforced since May 28, 2025) removes apps that carry `READ_MEDIA_IMAGES`/`READ_MEDIA_VIDEO` without broad-access core functionality ‚Äî a save-only app does not qualify. Saving your own file needs **no read permission** on modern Android (MediaStore write path). Mitigation: set the plugin's `granularPermissions` to the minimum (and/or `android.blockedPermissions` in app config for the READ_MEDIA_* entries) and rely on write-only save; verify the save path on an API 33+ device in M3. https://support.google.com/googleplay/android-developer/answer/14115180 ; https://support.google.com/googleplay/android-developer/answer/15800983
- The runtime `requestPermissionsAsync` also accepts `granularPermissions: ['photo']` (effective Android 13+). https://docs.expo.dev/versions/latest/sdk/media-library/

**expo-sharing ‚Äî VERIFIED, matches DESIGN.md**
- `expo-sharing ~57.0.2` (SDK 57). `Sharing.shareAsync(localFileUri, { mimeType: 'image/png', UTI, dialogTitle })` opens the native share sheet on iOS + Android with a local file (view-shot's `tmpfile` URI works directly). No permissions needed. Cannot pre-select a target app ‚Äî it is the general sheet, which is exactly what ¬ß10 assumes for V1. https://docs.expo.dev/versions/latest/sdk/sharing/

## 3. Instagram Stories direct share (V1.5 item)

- **Still alive in 2026.** Meta's "Sharing to Stories" docs are current: iOS = `instagram-stories://share?source_application=[appID]` + pasteboard keys `com.instagram.sharedSticker.backgroundImage` (JPG/PNG, min 720x1280), `.stickerImage`, `.backgroundTopColor/BottomColor`; Android = intent `com.instagram.share.ADD_TO_STORY` with `source_application` extra, content-URI data + `FLAG_GRANT_READ_URI_PERMISSION`. **A Facebook/Meta App ID is mandatory since January 2023** (share fails without it); no app review required, no deprecation notice on the page. https://developers.facebook.com/docs/instagram-platform/sharing-to-stories/
- 2025‚Äì2026 Meta deprecations (Basic Display API EOL Dec 2024, Graph API legacy-object migration Apr 2025, insights metric removals, Feb 2026 DM share-attachment change) **did not touch** the Stories intent/pasteboard mechanism ‚Äî it's an on-device app-to-app handoff, not a Graph API.
- **react-native-share v12.3.1** (actively maintained, 2026) supports it: `Share.shareSingle({ social: Share.Social.INSTAGRAM_STORIES, appId, backgroundImage, stickerImage, backgroundTopColor, backgroundBottomColor, attributionURL })`, iOS + Android both. It ships an **Expo config plugin** that adds `LSApplicationQueriesSchemes` (iOS) and the `<queries>` package-visibility entries (Android 11+) ‚Äî requires a dev build/prebuild, which the project already uses. Known open quirks are mostly around **video** to Stories on Android (#1345, #1464); static images (our case) are the reliable path. https://react-native-share.github.io/react-native-share/docs/share-single ; https://react-native-share.github.io/react-native-share/docs/install ; https://www.npmjs.com/package/react-native-share

## 4. Custom fonts (expo-font) in captured views

- No systemic incompatibility between custom fonts and view-shot ‚Äî the capture renders whatever the native view shows. The pitfalls are all **timing/config**:
  1. **Runtime loading race:** if fonts load via `useFonts`/`loadAsync`, a capture before `fontsLoaded` bakes the system fallback font into the image. Gate rendering/capture of the card on `fontsLoaded` (and on the card's `onLayout`).
  2. **Prefer build-time embedding:** the expo-font **config plugin** embeds fonts natively (recommended by Expo docs as more efficient than `useFonts`) ‚Äî fonts are then always available, eliminating the race entirely. Gotcha: natively embedded fonts are referenced by **PostScript name on iOS** vs file/family name on Android ‚Äî normalize `fontFamily` accordingly. https://docs.expo.dev/develop/user-interface/fonts/
  3. Use OTF/TTF only (other formats need Metro config and can crash); prefer OTF where both exist. There is an open Android issue about runtime-loaded TTFs falling back to the default font in some setups (expo/expo#33673) ‚Äî one more reason to use the config plugin for Fraunces/Literata. https://github.com/expo/expo/issues/33673
